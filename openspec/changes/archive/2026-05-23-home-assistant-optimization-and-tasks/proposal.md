# Home Assistant Optimization and Tasks

## Why

Home Assistant is now deployed declaratively with Authentik OIDC, but the repository does not yet contain a complete declarative inventory of automations, scripts, scenes, blueprints, or dashboards. This change proposes an optimization and maintenance plan that turns the current minimal platform configuration into an auditable Home Assistant configuration workflow.

## What Changes

- Audit the current committed Home Assistant configuration surface in `src/roles/nixos/files/etc/nixos/modules/home-assistant.nix`.
- Identify that `configuration.yaml` is generated by Nix, while `automations.yaml`, `scripts.yaml`, `scenes.yaml`, blueprints, and custom dashboards are referenced or requested but not committed in this repository.
- Add a declarative Home Assistant configuration structure for managed automations, scripts, scenes, dashboard assets, and helper definitions where appropriate.
- Add validation tasks to compare live `/var/lib/homeassistant` files against repository-managed files before importing or replacing runtime state.
- Review opportunities for official integrations before HACS or custom integrations, with existing `hass-oidc-auth` treated as a documented exception.
- Plan optimization work for onboarding state, OIDC, reverse proxy trust, observability, dashboard organization, and future local protocols such as Matter or Thread.

## Impact

- Affects `src/roles/nixos/files/etc/nixos/modules/home-assistant.nix` and any new repository-managed Home Assistant YAML/dashboard files added under `src/roles/nixos/files/etc/nixos/home-assistant/`.
- May require one-time migration of live Home Assistant runtime files from `/var/lib/homeassistant` into source control after review.
- Does not introduce new secrets, new public routes, or new non-official integrations during the proposal phase.
- Any later implementation must continue to use the existing Ansible/NixOS deployment workflow.
